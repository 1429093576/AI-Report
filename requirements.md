# AI舆情分析日报系统（Daily AI Insight Engine）需求说明

## 1. 项目背景

本项目旨在构建一个最小可用版本（MVP）的 AI 舆情分析日报系统，用于从每日新闻、技术发布、社区讨论等信息中提取结构化洞察，并生成可读的分析报告与可视化结果。

系统主要服务于以下场景：

- AI 行业趋势分析
- 舆情监测与风险预警
- 信息快速理解与决策辅助
- 技术、产品、资本、政策方向的信息归纳

本项目不仅要求完成一个可运行的 MVP，还要求体现 Harness Engineering 思想，通过开源或自建 Harness 框架沉淀可复用的 Agent 工作流、Prompt、Hook、Skill 等工程资产。

## 2. 项目目标

建设一个能够完成“数据获取 -> 数据清洗 -> AI 相关性准入 -> 结构化抽取 -> 分析生成 -> 可视化展示 -> 输出日报”的 MVP 系统。

最终系统应能够基于报告日当天的 AI 相关新闻或信息，生成一份完整的“AI 分析日报”，并保留完整处理过程与设计说明，便于后续扩展、复现和评估。

## 3. 使用对象

- AI 行业研究人员
- 产品经理与战略分析人员
- 舆情监测人员
- 投资、咨询、媒体分析相关人员
- 需要快速理解 AI 行业动态的业务决策者

## 4. 核心范围

### 4.1 必须完成

- 构建一个最小可用版本（MVP）的 AI 舆情分析日报系统。
- 自主确定并说明数据源。
- 获取至少 10 到 20 条报告日当天的 AI 相关信息。
- 日报输入必须严格限制为 `run_date` 对应报告日当天发布的信息，不能以“近 7 天”或“最近若干条”替代“当天”。
- 必须提供 AI 科技相关性准入机制，确保只有属于 AI 科技日报范围的条目进入结构化抽取、分析和报告。
- 每条数据至少包含：
  - 标题
  - 正文内容或摘要
  - 来源
  - 发布时间
- 设计结构化数据 Schema。
- 基于 Schema 对新闻信息进行结构化抽取。
- 生成一份完整的 AI 分析日报。
- 日报必须包含：
  - 信息结构化结果
  - 分析报告
  - 可视化展示
- 保留所有 Prompt。
- 保留原始数据文件。
- 提供数据来源说明和数据选择理由。
- 提供系统设计说明文档。
- 提供至少 1 份完整的输出结果示例。
- 将项目代码、数据、Prompt、输出样例和说明文档提交到 GitHub 仓库。
- 研究 Harness Engineering 思想，并在项目中体现。
- 使用开源或自建 Harness 框架沉淀：
  - `agent.md`
  - Hook
  - Skills
  - Prompt 模板或 Prompt 资产
- 至少包含一个专门用于 AI 科技相关新闻判定的 Skill，并说明它如何在主流程中参与准入判断。
- 实现轻量级主题索引记忆机制（Topic-Indexed Memory）：
  - 必须能按标准化 `topic` 保存已校验通过的结构化新闻。
  - 必须能在日报分析前检索同主题历史上下文。
  - 必须能说明记忆机制如何辅助短期趋势分析、上下文增强和运行复现。

### 4.2 可选完成

- 数据源支持中英文混合。
- 接入更多类型的信息源，例如：
  - 科技媒体
  - 官方博客
  - GitHub Releases
  - arXiv
  - Twitter/X
  - Reddit
  - 知乎热榜
  - Google News
  - Hacker News
  - Product Hunt
- 实现自动化爬虫或 API 数据采集。
- 增加风险提示或机会提示模块。
- 增加多维度可视化，例如：
  - 热点主题分布
  - 来源分布
  - 情绪倾向分布
  - 公司或产品提及频次
  - 技术方向趋势图
- 支持日报导出为 Markdown、HTML、PDF 或 PPT。
- 增加自动化任务调度，例如每日定时运行。
- 增加结构化结果校验机制。
- 增加人工审核或修订入口。
- 增加多 Agent 协作流程，例如采集 Agent、抽取 Agent、分析 Agent、校验 Agent。

### 4.3 禁止事项

- 不允许将原始数据一次性全部丢给 AI，让 AI 直接生成全部最终结果。
- 不允许只调用现成 AI 产品生成结果后截图提交。
- 不允许只做简单摘要拼接。
- 不允许只提交分析报告而不提供结构化数据模型。
- 不允许没有数据清洗、分批处理、校验或其他处理逻辑。
- 不允许缺少 Prompt 记录。
- 不允许缺少原始数据文件。
- 不允许缺少数据来源说明。
- 不允许缺少系统设计说明。
- 不允许没有完整的 AI 分析日报输出样例。

## 5. 数据需求

### 5.1 数据来源要求

数据源由项目实施方自主寻找和确定，可参考以下方向：

| 类型 | 示例来源 | 说明 |
| --- | --- | --- |
| 科技媒体 | TechCrunch、The Verge、机器之心、量子位 | 行业动态报道 |
| 官方渠道 | 公司博客、GitHub Releases、arXiv | 技术发布信息 |
| 社交媒体 | Twitter/X、Reddit、知乎热榜 | 舆论讨论热点 |
| 聚合平台 | Google News、Hacker News、Product Hunt | 综合信息源 |

不同类型数据源应采用不同处理策略，不能把所有来源都当作普通新闻正文处理：

- 科技媒体和公司博客应优先获取完整原文正文；RSS 只用于发现文章列表，不能把 RSS 摘要视为完整正文。
- GitHub Releases、arXiv、Product Hunt 等结构化平台应优先使用官方 API，保留发布正文、论文摘要、产品描述和平台元数据。
- Twitter/X、Reddit、知乎热榜、Hacker News 等社交或社区来源应作为舆论讨论和热度信号处理，保留互动指标、排名、评论数和外链；如果存在外链新闻，再单独抓取外链正文。
- Google News 等聚合平台应作为发现入口，必须尽量解析到原始媒体 URL，再按原始来源抓取正文；聚合平台摘要不能替代原文。
- 对需要凭证、授权、反爬或页面结构不稳定的数据源，应先登记为 disabled，并在文档中说明启用条件和降级策略。

### 5.2 数据字段要求

原始数据文件中，每条信息至少应包含：

- `title`：标题
- `content` 或 `summary`：正文内容或摘要
- `source`：来源
- `published_at`：发布时间

建议额外包含：

- `url`：原文链接
- `language`：语言
- `source_type`：来源类型
- `author`：作者或发布主体
- `collected_at`：采集时间
- `canonical_url`：去重后的原始来源链接
- `discovery_source`：发现该条信息的来源，例如 Google News 或 Hacker News
- `content_source`：正文来源，例如 `article_html`、`rss_feed`、`github_release_body`、`arxiv_summary`、`social_post`
- `engagement`：社交或聚合平台的互动指标，例如 score、comments、rank、votes

### 5.3 数据量要求

- 必须获取至少 10 到 20 条报告日当天的 AI 相关新闻或信息。
- 数据应尽量覆盖不同来源和不同主题，避免全部来自单一渠道。
- 当某些来源只能返回最近窗口数据时，系统仍必须在进入抽取前按 `run_date` 和报告时区执行“仅保留当天新闻”的硬过滤。

### 5.4 数据选择理由

说明文档中必须解释：

- 为什么选择这些数据源。
- 数据源的可信度如何。
- 数据源覆盖了哪些 AI 信息类型。
- 数据是否存在偏差，例如媒体报道偏差、社区讨论噪声、语言偏差等。

## 6. 结构化抽取需求

### 6.1 Schema 设计要求

必须设计结构化数据模型，不能只做摘要。

Schema 需要体现对新闻信息的抽取和整理能力，建议包含以下字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 新闻唯一标识 |
| `title` | 新闻标题 |
| `source` | 新闻来源 |
| `published_at` | 发布时间 |
| `source_type` | 来源类型 |
| `language` | 语言 |
| `topic` | 主题分类 |
| `entities` | 涉及公司、产品、人物、机构 |
| `event_type` | 事件类型，例如产品发布、融资、政策、研究进展、争议事件 |
| `summary` | 结构化摘要 |
| `key_points` | 关键信息点 |
| `sentiment` | 舆情倾向，例如正面、中性、负面、争议 |
| `impact_scope` | 影响范围，例如技术、产业、资本、政策、用户 |
| `importance_score` | 重要性评分 |
| `risk_level` | 风险等级 |
| `opportunity_level` | 机会等级 |
| `evidence` | 支撑判断的原文证据或片段 |

如需支持 AI 相关性准入审计，建议额外包含以下字段或等价信息：

| 字段 | 说明 |
| --- | --- |
| `is_ai_related` | 是否通过 AI 科技相关性准入 |
| `relevance_score` | 相关性评分 |
| `relevance_reason` | 相关性判断理由 |
| `relevance_evidence` | 相关性判断证据 |

### 6.2 Schema 设计说明

必须在文档中说明 Schema 的设计思路，包括：

- 为什么设计这些字段。
- 字段如何服务后续分析。
- 哪些字段用于排序重要事件。
- 哪些字段用于趋势判断。
- 哪些字段用于风险或机会识别。

### 6.3 抽取处理要求

必须体现处理逻辑，例如：

- 数据清洗
- 去重
- 报告日当天硬过滤
- AI 科技相关性准入判断
- 按来源或主题分批处理
- 使用 Prompt 进行结构化抽取
- 对抽取结果进行格式校验
- 对异常结果进行重试或人工修正
- 合并批次结果

## 7. 分析报告需求

生成的“AI 分析日报”必须包含以下内容。

### 7.1 今日 AI 领域主要热点

输出 Top 3 到 5 个重要事件。

每个热点应包含：

- 事件标题
- 来源或相关信息
- 重要性判断
- 关键原因
- 可能影响

### 7.2 重要事件深度总结

对关键事件进行背景与影响分析。

应包含：

- 事件背景
- 当前进展
- 涉及主体
- 对行业、技术、产品、资本或政策的影响
- 后续值得关注的问题

### 7.3 趋势判断

基于结构化数据进行趋势洞察，至少覆盖以下方向中的若干项：

- 技术趋势
- 应用趋势
- 政策趋势
- 资本趋势
- 开源生态趋势
- 舆论风险趋势

趋势判断必须有逻辑支撑，不能空泛描述。

日报分析所使用的主数据集必须全部来自报告日当天；历史上下文只能作为趋势参照，不能混入“今日热点”主输入。

### 7.4 风险或机会提示

该部分为可选，但建议实现。

可识别：

- 潜在监管风险
- 模型安全或数据安全风险
- 舆论争议风险
- 商业化机会
- 投资或产品机会
- 开源生态机会

## 8. 可视化需求

系统输出中必须包含可视化内容，形式不限。

### 8.1 必须满足

- 可视化必须能够清晰传达信息。
- 可视化应基于结构化结果生成，而不是独立手工绘制。
- 可视化内容应与日报分析结论相互支撑。

### 8.2 可视化形式示例

可选择以下一种或多种：

- 主题分类柱状图
- 来源分布饼图
- 重要性评分排序图
- 情绪分布图
- 风险等级分布图
- 技术关键词词云
- 时间线图
- 公司或产品提及频次图

### 8.3 技术栈

技术栈不限，可使用：

- Python + Matplotlib / Plotly / Altair
- JavaScript + ECharts / D3.js
- Markdown Mermaid
- HTML 可视化页面
- Jupyter Notebook

## 9. 轻量级记忆机制要求

### 9.1 机制定位

MVP 必须实现轻量级主题索引记忆机制（Topic-Indexed Memory），用于在单次或短周期批处理场景中保存和检索历史上下文。

该机制的目标是：

- 为趋势分析提供近 N 天同主题历史参照。
- 为日报生成提供可解释的上下文增强。
- 为 Harness 流程提供可追踪、可复现的状态依据。

该机制不承担完整长期知识库、向量检索系统或实体关系图谱职责。

### 9.2 存储形式

记忆数据采用 JSON 文件存储，推荐文件路径为：

```text
memory/topic_index.json
```

存储结构以标准化 `topic` 作为一级 Key，将已校验通过的结构化新闻按主题归档。每条新闻节点建议保留：

- `date`：新闻日期或归档日期
- `id`：结构化新闻 ID
- `title`：新闻标题
- `summary`：结构化摘要
- `key_entities`：关键实体
- `source`：来源
- `importance_score`：重要性评分
- `url` 或 `content_hash`：用于去重的依据

### 9.3 Hook 交互要求

记忆机制必须与 Harness 流程中的 Hook 协作，而不是作为孤立文件存在。

- Pre-Hook：在日报分析前触发，根据当日新闻命中的 `topic`，从 `memory/topic_index.json` 中读取近 N 天同主题历史新闻，作为趋势分析上下文注入后续 Prompt。
- Post-Hook：在日报生成完成后触发，将当日校验通过的结构化新闻追加写入 `memory/topic_index.json`。

### 9.4 约束规则

记忆机制必须包含以下约束：

- 写入前必须基于 `url`、`content_hash` 或稳定 `id` 进行去重。
- 检索时必须设置时间窗口，例如近 N 天。
- 检索时必须设置每个主题的最大召回数量，避免上下文过长。
- 检索结果必须区分历史新闻与当日输入新闻，避免重复注入上下文。
- 只有通过 Schema 校验的结构化新闻才允许写入记忆。
- 低置信度或异常数据应被标记，不能直接作为可靠趋势依据。

### 9.5 MVP 不包含

MVP 阶段不要求实现以下能力：

- 向量数据库
- 图数据库
- 实体关系图谱
- 复杂时间衰减算法
- 长期趋势追踪数据库
- 多月或多年级别的历史知识库

长期历史日报对比和趋势追踪数据库可作为后续扩展能力，不属于 MVP 必须范围。

## 10. Harness Engineering 要求

### 10.1 必须体现

项目需要研究并体现 Harness Engineering 的思想，即通过工程化方式管理 AI Agent 的输入、输出、流程、工具、校验和复用资产。

### 10.2 需要沉淀的内容

必须在项目中沉淀以下内容：

- `agent.md`：定义 Agent 角色、目标、约束、输入输出格式。
- Prompt 文件：保留结构化抽取、分析生成、校验等 Prompt。
- Hook：用于流程前后处理、校验、日志记录或失败重试。
- Skills：定义可复用能力，例如新闻抽取、事件分类、风险识别、日报生成。
- Skills 中必须至少有一个专门处理 AI 科技相关性准入的能力包，例如 `ai_news_relevance/`，用于沉淀判断标准、边界案例、示例和校验资源。
- 记忆机制：通过 Pre-Hook 和 Post-Hook 管理历史上下文读取与写入。
- Checkpointer：记录关键步骤执行前的稳定文件状态，支持失败后的文件回滚和异常追踪。

### 10.3 记忆机制与 Hook 协作

Harness 需要将记忆机制作为上下文管理能力纳入流程：

- 分析前由 Pre-Hook 读取同主题历史上下文。
- 分析后由 Post-Hook 写入当日已校验新闻。
- Hook 需要记录读取和写入的主题、数量、时间窗口和异常情况。
- Agent 使用历史上下文时，必须区分“历史事实”和“今日新增信息”。

### 10.4 Checkpointer 与失败回滚要求

Harness 必须包含轻量 Checkpointer 和失败回滚能力，避免任一步骤失败后留下不一致的中间产物。

必须满足以下要求：

- 在关键 Pipeline 步骤执行前记录 checkpoint。
- checkpoint 需要覆盖本步骤可能改写的 artifact，例如 `data/`、`outputs/`、`logs/validation_report.json`、`memory/topic_index.json`。
- 如果步骤失败，必须恢复到该步骤开始前的稳定文件状态。
- 失败信息、回滚结果、受影响文件、`run_id`、`step_name` 必须写入 `state/run_manifest.json` 或等价 manifest。
- MVP 先支持“回滚 + 重新完整运行”，暂不要求从任意步骤续跑。
- 如果无法完成回滚，必须显式标记失败，不能静默继续生成日报。
- 回滚逻辑应由 Harness 层统一处理，业务模块不应在 `collect`、`extract`、`analyze`、`generate_report` 等步骤中各自实现回滚。

### 10.5 建议实现方式

可以使用：

- 开源 Agent/Harness 框架
- 自建轻量 Harness 框架
- CLI 脚本 + 配置文件 + Prompt 模板 + Hook 机制

MVP 阶段建议优先选择轻量实现，保证流程清晰、可复现、可扩展。

## 11. 核心流程

建议系统流程如下：

1. 数据采集
2. 原始数据保存
3. 数据清洗与去重
4. 报告日当天硬过滤
5. AI 科技相关性准入判断
6. 分批结构化抽取
7. Schema 校验
8. 结构化结果合并
9. 重要性评分与排序
10. 记忆检索：Pre-Hook 根据当日命中主题读取近 N 天同主题历史上下文
11. 主题、趋势、风险分析
12. 可视化生成
13. 日报生成
14. 输出样例归档
15. 记忆写入：Post-Hook 将当日校验通过的结构化新闻写入主题索引记忆

| `agent.md` | 必须 | Agent 定义与运行约束 |
| Hook 文件或说明 | 必须 | 体现流程控制或校验机制 |

| 自动化爬虫 | 可选 | 可用静态数据替代 |
| 多 Agent 协作 | 可选 | MVP 可简化 |
| 定时任务 | 可选 | 后续扩展项 |

## 13. 说明文档必须包含

说明文档是核心考察项，必须覆盖以下内容：

- 数据源说明
  - 来源

  - 如何保证只纳入报告日当天新闻
  - 如何判定一条新闻属于 AI 科技日报范围

  - 关键技术决策

  - Prompt 设计
  - 分批处理方式

## 14. MVP 验收标准

MVP 可被认为完成的最低标准如下：

- GitHub 仓库中包含完整项目文件。
- 至少有 10 条报告日当天的 AI 相关数据。
- 原始数据字段完整。
- 有明确的数据来源和选择理由。
- 有结构化 Schema 及设计说明。
- 有结构化抽取结果，不只是摘要。
- 有分批处理、清洗、校验或类似处理逻辑。
- 有明确的“报告日当天”硬过滤逻辑。
- 有明确的 AI 科技相关性准入逻辑，而不是仅依赖数据源名称或 topic 回填。
- 有至少 1 份完整 AI 分析日报。
- 日报包含 Top 事件、深度总结、趋势判断。
- 日报中包含至少 1 个可视化结果。
- 保留所有 Prompt。
- 包含 Harness Engineering 相关内容：
  - `agent.md`
  - Hook
  - Skills
  - AI 科技相关性 Skill
  - 轻量 Checkpointer
  - 失败回滚机制
  - 回滚记录与异常可追踪
- 包含轻量级主题索引记忆机制：
  - 能展示 `memory/topic_index.json` 或样例文件结构。
  - 能说明记忆检索与写入逻辑。
  - 能说明去重、时间窗口和每主题最大召回数量。
  - 能在日报或说明文档中体现历史上下文如何辅助趋势分析。
  - 能说明为什么 MVP 不使用向量数据库或图数据库。
- 说明文档能够解释完整系统流程和关键设计决策。

## 16. 关键风险与注意事项

- 数据质量会直接影响分析质量，需要优先保证来源可信和字段完整。
- 社交媒体数据噪声较高，应考虑过滤、去重和可信度判断。
- “最近窗口数据”不等于“日报当天数据”，必须在主流程前段做显式日期过滤。
- “主题分类像 AI”不等于“适合纳入 AI 日报”，必须有独立的相关新闻准入判断。
- AI 抽取结果可能存在幻觉，需要通过 Schema 校验和证据字段降低风险。
- 趋势判断必须基于结构化数据，避免主观空泛结论。
- Prompt 需要版本化保存，便于复现和评估。
- MVP 不应过度追求复杂爬虫，静态数据也可以满足任务要求，但必须体现处理逻辑。
- Harness Engineering 不应停留在概念描述，需要通过文件、流程或代码体现。
- 记忆机制应保持轻量，避免在 MVP 阶段扩展为复杂知识库、图数据库或向量检索系统。

## 17. 优先级建议

### P0：必须优先完成

- 原始数据整理
- Schema 设计
- 结构化抽取
- Prompt 留存
- 日报生成
- 可视化输出
- README / 系统说明文档
- `agent.md`、Hook、Skills 基础沉淀
- 轻量级主题索引记忆机制
- 轻量 Checkpointer
- 失败回滚机制
- 回滚记录与异常可追踪

### P1：建议完成

- 数据清洗与去重
- 抽取结果校验
- 重要性评分
- 风险和机会提示
- 多数据源覆盖
- HTML 或 Markdown 格式美化

### P2：后续扩展

- 自动爬虫
- 定时任务
- 多 Agent 工作流
- Web UI
- 历史日报对比
- 趋势追踪数据库
- 自动发布日报

长期记忆能力、历史日报对比和趋势追踪数据库不属于 MVP 必须范围，应作为后续扩展规划。
